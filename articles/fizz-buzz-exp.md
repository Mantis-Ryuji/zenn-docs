## はじめに

先日、面白い記事を読みました。**5年前** に公開された [tokoroten](https://zenn.dev/tokoroten) さんの記事です。テーマは、FizzBuzzを機械学習で解くというもの（なぜか For you にあった）。普段座学ばかりなのでたまには実験してみます。

@[card](https://zenn.dev/tokoroten/articles/c311cf6e3fc8ac)

一見すると、FizzBuzz を機械学習で解くことには何の意味もなさそうです。しかし、これを単に「解けるかどうか」の問題としてではなく、**ニューラルネットワークが有限の訓練標本から剰余規則を獲得し、桁数の異なる整数へ外挿できるのか** という問題として捉えると、かなり面白い題材に見えてきます。

FizzBuzz を機械学習で解く試み自体は、案外いろいろなところで行われているようです。精度100%を謳う例もかなりあります。しかし、それは本当にモデルが **「剰余規則を獲得した」** と言えるのでしょうか。少なくとも、そこまで簡単なタスクではないはずです。
（そもそもそんなタスクとして捉えていない...）

そこで本記事では、この点を検証してみます。

まず、本記事では特徴量として $\bmod$ を与えることはしません。**これは実質的に答えを渡しているのと同じだからです。** FizzBuzz の本質は、入力された整数から3や5で割り切れるかを判定することにあります。したがって、剰余を特徴量として与えてしまうと、ニューラルネットワークが規則を獲得したのか、単に与えられた答えを読み取ったのかが区別できなくなります。

また、以下の記事も興味深いです。

@[card](https://qiita.com/tanuk1647/items/30a03a9888d39d307e73)

この記事では、**「FizzBuzzのように明確な周期をもったデータに対してLSTMを適用すれば、100点も取れるし、外挿にも成功する」** と書かれています。ただし、ここで用いられている設定は、FizzBuzz の出力ラベル列を周期系列として扱い、過去15ステップのラベルから次のラベルを予測するというものです。

この設定では、入力は整数そのものではなく、すでに FizzBuzz 規則によって変換されたラベル列です。したがって、仮に 10000 ステップ先まで正しく生成できたとしても、それは **「整数表現から剰余規則を学習し、未知の整数へ外挿した」** ことを意味するわけではありません。

:::details 周期15を与えると、何が問題なのか？
該当記事では、LSTM に過去の出力列を入力し、次の FizzBuzz ラベルを予測する設定になっています。また、`num_steps` について「15個のデータで1サイクルなので、理論上、num_stepsは15で良いはずです」と説明されています。

しかし、この時点で本記事が考えたい問題とはすでに別物です。FizzBuzz の周期が15であることは、3と5の最小公倍数から出てくる本質的な構造です。これを人間が先に見抜いて、`num_steps=15` としてモデルに与えてしまうなら、モデルはもはや剰余規則を自分で獲得する必要がありません。

入力されているのも整数そのものではなく、すでに FizzBuzz 規則によって変換された答えの周期列です。つまり、モデルが見ているのは **「整数」** ではなく **「答えの周期パターン」** です。過去15個のラベルを見れば、周期中の位置は分かるので、次のラベルも決まります。

したがって、この設定で精度100%を達成したとしても、それは整数表現から FizzBuzz の規則を学習したことを意味しません。15周期の答えのパターンをなぞっているだけです。
:::

だから、そもそもそんなタスクとして捉えていないって？
...聞こえません。

これは系列の周期性を利用した予測であり、本記事で考えたい外挿とは性質が異なります。

他にも面白い実験があり、強化学習で FizzBuzz を解くというものもあります。

@[card](https://qiita.com/ymg_aq/items/d276357823f5b2ce46c5)

この記事では、FizzBuzz を二人零和有限確定完全情報ゲームとして扱い、自己対戦によって 1 から 100 までを正しく出力できる policy を学習しています。発想としてはとても面白いです。

ただし、前の記事と同様に、これも本記事で考えたい「剰余規則の獲得」とは少し異なります。この設定では、ゲームは 1 から **順に** 進み、100 まで到達したら終了します。また、入力も **整数そのものではなく直前10個の履歴** です。序盤は履歴がそろっていないため、初期状態を含む数手分のパターンを覚える必要がありますが、いったん正しい履歴が10個そろえば、FizzBuzz の出力列は周期15なので、直前10個の履歴から次のラベルを決定できます。

:::details 直前10個の履歴から次のラベルを決定する
FizzBuzz の出力列は周期15です。1周期を

$$
\text{(N,N,F,N,B,F,N,N,F,B,N,F,N,N,FB)}
$$

とします。ここで、$\text{N}$ は Number、$\text{F}$ は Fizz、$\text{B}$ は Buzz、$\text{FB}$ は FizzBuzz です。

各位相から長さ10の履歴を取り出すと、次のようになります。

| 位相 | 直前10個の履歴      | 次のラベル |
| -: | -------------------- | ----- |
|  1 | $\text{(N, N, F, N, B, F, N, N, F, B)}$  | $\text{N}$     |
|  2 | $\text{(N, F, N, B, F, N, N, F, B, N)}$  | $\text{F}$     |
|  3 | $\text{(F, N, B, F, N, N, F, B, N, F)}$  | $\text{N}$     |
|  4 | $\text{(N, B, F, N, N, F, B, N, F, N)}$  | $\text{N}$     |
|  5 | $\text{(B, F, N, N, F, B, N, F, N, N)}$  | $\text{FB}$    |
|  6 | $\text{(F, N, N, F, B, N, F, N, N, FB)}$ | $\text{N}$     |
|  7 | $\text{(N, N, F, B, N, F, N, N, FB, N)}$ | $\text{N}$     |
|  8 | $\text{(N, F, B, N, F, N, N, FB, N, N)}$ | $\text{F}$     |
|  9 | $\text{(F, B, N, F, N, N, FB, N, N, F)}$ | $\text{N}$     |
| 10 | $\text{(B, N, F, N, N, FB, N, N, F, N)}$ | $\text{B}$     |
| 11 | $\text{(N, F, N, N, FB, N, N, F, N, B)}$ | $\text{F}$     |
| 12 | $\text{(F, N, N, FB, N, N, F, N, B, F)}$ | $\text{N}$     |
| 13 | $\text{(N, N, FB, N, N, F, N, B, F, N)}$ | $\text{N}$     |
| 14 | $\text{(N, FB, N, N, F, N, B, F, N, N)}$ | $\text{F}$     |
| 15 | $\text{(FB, N, N, F, N, B, F, N, N, F)}$ | $\text{B}$     |

この15個の履歴はすべて異なります。したがって、正しい FizzBuzz 列上で直前10個の履歴がそろっていれば、その履歴から周期中の位置を一意に特定できます。位相が決まれば次のラベルも決まるため、直前10個の履歴から次の FizzBuzz ラベルを決定できます。

つまり、必要なのは **「序盤の初期履歴パターン」** と **「周期15上の履歴パターン」** を覚えることであり、**整数から剰余を計算することではありません。** したがって、1〜100 を正しく出力できたとしても、それは履歴パターンを利用して限定されたゲームを攻略したという話であって、「整数表現から剰余規則を **マスター** した」とは言えません。

## 1. 問題設定

本記事では、FizzBuzzを4クラス分類問題として扱います。入力は整数 $n$ であり、出力は以下の4クラスのいずれかです。

| クラス | 条件                    | ラベル        |
| --: | --------------------- | ---------- |
|   0 | $n$ が3でも5でも割り切れない     | `Number`   |
|   1 | $n$ が3で割り切れ、5では割り切れない | `Fizz`     |
|   2 | $n$ が5で割り切れ、3では割り切れない | `Buzz`     |
|   3 | $n$ が15で割り切れる         | `FizzBuzz` |

ここで重要なのは、学習時に `n % 3` や `n % 5` のような剰余特徴量を与えないことです。本記事で見たいのは、モデルが明示的に与えられた剰余を読み取れるかではなく、整数から剰余規則を学習し、未知の桁数へ外挿できるかです。

具体的には、学習データには **1桁から5桁までの整数** のみを用います。一方、テストデータは **6桁、7桁、8桁** に分けて評価します。これにより、学習時に見た桁数の範囲内での性能ではなく、桁数が増えたときに規則が保たれるかを確認します。

| 分割          |   桁数 |                          整数範囲 | 目的        |
| ----------- | ---: | ----------------------------: | --------- |
| Train       | 1〜5桁 |          $1 \leq n \leq 99999$ | 剰余規則の学習   |
| Test-6 digit |   6桁 |     $100000 \leq n \leq 999999$ | 1段階外挿の評価  |
| Test-7 digit |   7桁 |   $1000000 \leq n \leq 9999999$ | 2段階外挿の評価 |
| Test-8 digit |   8桁 | $10000000 \leq n \leq 99999999$ | 3段階外挿の評価  |

この設定では、テストデータは学習データと数値範囲が完全に分離されています。したがって、単に近傍の整数に対する補間ができているだけでは高い性能は得られません。特に8桁テストでは、学習範囲から大きく離れた整数に対して FizzBuzz 規則を適用できるかが問われます。

また、本記事ではモデルサイズを変えたときに性能がどのように変化するかも検証します。

| モデルサイズ    | 目的                     |
| ------ | ---------------------- |
| `small`  | 低容量モデルでも剰余規則を獲得できるかを見る |
| `medium` | 標準的な容量での外挿性能を見る        |
| `large`  | 容量を増やすことで外挿性能が改善するかを見る |

ここで重要なのは、モデルサイズを大きくしたときに訓練精度だけでなく、6桁・7桁・8桁への外挿性能が改善するかを見ることです。もしモデルが単に 1 から 99999 までの対応関係を記憶しているだけなら、モデルサイズを大きくしてもテスト性能は改善しないか、むしろ訓練範囲への過適合によって外挿性能が悪化する可能性があります。

一方で、モデル容量を増やすことで剰余規則に対応する内部表現が獲得されるのであれば、6桁、7桁、8桁に対する性能も改善するはずです。したがって、本実験では **モデルのスケーリングが FizzBuzz 規則の獲得に寄与するのか** も確認します。

## 2. 評価指標

評価には、Accuracyだけでなく、Precision、Recall、F1-scoreも用います。FizzBuzzはクラス分布が均等ではなく、特に `FizzBuzz` は15個に1個しか出現しないため、Accuracyだけでは少数クラスの失敗を見落とす可能性があります。

特に重視するのは **Macro F1** です。FizzBuzzでは `Number` クラスが最も多く、`FizzBuzz` クラスが最も少ないため、モデルが多数派クラスに偏ってもAccuracyはある程度高く見えてしまいます。Macro F1を見ることで、各クラスを同じ重みで評価し、少数クラスである `FizzBuzz` を正しく扱えているかを確認できます。

また、テスト結果は 6桁、7桁、8桁で個別に集計します。これにより、単に「外挿できたか」だけでなく、桁数が増えるにつれて性能がどのように劣化するかを確認します。

:::details 評価指標のおさらい
ここでは、本記事で用いる分類評価指標を簡単に整理しておきます。
FizzBuzz分類は、次の4クラス分類問題として扱います。

$$
\mathcal{Y}=\{\mathrm{Number}, \mathrm{Fizz}, \mathrm{Buzz}, \mathrm{FizzBuzz}\}
$$

サンプル数を $N$、正解ラベルを $y_i$、予測ラベルを $\hat{y}_i$ とします。ここで $i=1,\ldots,N$ です。

### Accuracy

Accuracy は、全サンプルのうち正しく分類できた割合です。

$$
\mathrm{Accuracy}=\frac{1}{N}\sum_{i=1}^{N}\mathbf{1}(y_i=\hat{y}_i)
$$

ここで、$\mathbf{1}(\cdot)$ は条件が真なら1、偽なら0を返す指示関数です。

Accuracy は全体の正解率を直感的に表しますが、クラス不均衡がある場合には注意が必要です。FizzBuzzでは `Number` が最も多く、`FizzBuzz` が最も少ないため、多数派クラスをよく当てるだけでもAccuracyは高く見える可能性があります。

### Precision

クラス $c$ に対する Precision は、モデルがクラス $c$ と予測したサンプルのうち、実際にクラス $c$ だった割合です。

$$
\mathrm{Precision}_c=\frac{\mathrm{TP}_c}{\mathrm{TP}_c+\mathrm{FP}_c}
$$

ここで、各記号は次のように定義します。

| 記号              | 意味                             |
| --------------- | ------------------------------ |
| $\mathrm{TP}_c$ | 正解が $c$ で、予測も $c$ だったサンプル数     |
| $\mathrm{FP}_c$ | 正解は $c$ ではないが、予測が $c$ だったサンプル数 |
| $\mathrm{FN}_c$ | 正解は $c$ だが、予測が $c$ ではなかったサンプル数 |

Precision は、「そのクラスだと予測したものがどれだけ信用できるか」を表します。たとえば `FizzBuzz` の Precision が低い場合、モデルは `FizzBuzz` と予測しすぎており、実際には別クラスのサンプルも多く混ざっていることを意味します。

### Recall

クラス $c$ に対する Recall は、実際にクラス $c$ であるサンプルのうち、モデルが正しくクラス $c$ と予測できた割合です。

$$
\mathrm{Recall}_c=\frac{\mathrm{TP}_c}{\mathrm{TP}_c+\mathrm{FN}_c}
$$

Recall は、「そのクラスをどれだけ取りこぼさずに拾えたか」を表します。たとえば `FizzBuzz` の Recall が低い場合、本来 `FizzBuzz` であるサンプルの多くを `Fizz`、`Buzz`、`Number` などに誤分類していることを意味します。

### F1-score

クラス $c$ に対する F1-score は、Precision と Recall の調和平均です。

$$
\mathrm{F1}_c=\frac{2 \cdot \mathrm{Precision}_c \cdot \mathrm{Recall}_c}{\mathrm{Precision}_c+\mathrm{Recall}_c}
$$

F1-score は、Precision と Recall の片方だけが高くても大きな値になりにくい指標です。したがって、「誤検出を抑えつつ、取りこぼしも少ないか」を見るために使えます。

たとえば、あるクラスについて Precision が高くても Recall が低い場合、モデルは慎重にそのクラスを予測しているものの、多くの正解サンプルを取りこぼしていることになります。逆に Recall が高くても Precision が低い場合、そのクラスを広く拾えてはいるものの、誤検出が多いことになります。

### Macro平均

Macro平均は、各クラスの指標を単純平均したものです。たとえば Macro F1 は次のように定義されます。

$$
\mathrm{Macro\ F1}=\frac{1}{|\mathcal{Y}|}\sum_{c \in \mathcal{Y}}\mathrm{F1}_c
$$

Macro平均では、各クラスが同じ重みで扱われます。そのため、サンプル数の少ない `FizzBuzz` も、サンプル数の多い `Number` と同じ重みで評価されます。
:::

## 3. モデル設計

ここで、FizzBuzz を少し別の角度から眺めます。

整数を10進数字列として扱うと、FizzBuzzは **「左から数字を読みながら内部状態を更新する問題」** と見なせます。たとえば、整数 $n=12345$ は次のような数字列として表現できます。

$$
n = 12345
\quad \longmapsto \quad
(d_1,d_2,d_3,d_4,d_5) = (1,2,3,4,5)
$$

このとき、先頭から順に数字を読んでいけば、現在までに読んだ部分整数の剰余を逐次的に更新できます。具体的に、法 $m$ に対する剰余を考えると、左から $t$ 桁目まで読んだ時点の剰余 $r_t^{(m)}$ は次のように更新されます。

$$
r_t^{(m)}=(10r_{t-1}^{(m)} + d_t) \bmod m
$$

ここで、$d_t$ は左から $t$ 番目の数字、$r_t^{(m)}$ は先頭から $t$ 桁まで読んだ整数を $m$ で割った余りです。

:::details 具体例: n=12345 が 3 で割り切れるかを逐次的に考えてみる
たとえば、$n=12345$ が 3 で割り切れるかを考えてみます。この場合は $m=3$ なので、次のように剰余状態を更新していきます。

|    読んだ数字 | ここまでの整数 | 更新式                              | 剰余 |
| -------: | ------: | -------------------------------- | -: |
| まだ読んでいない |       - | $r_0^{(3)}=0$                    |  $\color{red}{0}$ |
|        1 |       1 | $r_1^{(3)}=(10\cdot {\color{red}{0}}+1)\bmod 3$ |  $\color{tomato}{1}$ |
|        2 |      12 | $r_2^{(3)}=(10\cdot {\color{tomato}{1}}+2)\bmod 3$ |  $\color{orange}{0}$ |
|        3 |     123 | $r_3^{(3)}=(10\cdot {\color{orange}{0}}+3)\bmod 3$ |  $\color{green}{0}$ |
|        4 |    1234 | $r_4^{(3)}=(10\cdot {\color{green}{0}}+4)\bmod 3$ |  $\color{skyblue}{1}$ |
|        5 |   12345 | $r_5^{(3)}=(10\cdot {\color{skyblue}{1}}+5)\bmod 3$ |  $\color{magenta}{0}$ |

最後に得られた剰余は $r_5^{(3)}=0$ です。したがって、

$$
12345 \bmod 3 = 0
$$

であり、12345 は 3 で割り切れることが分かります。

重要なのは、この計算では整数全体を一度に見る必要がないという点です。左から1桁ずつ読みながら、現在の剰余状態だけを更新すれば、最後に $n \bmod 3$ を求めることができます。
:::

FizzBuzz では 3 で割り切れるか、5 で割り切れるかを判定すればよいので、少なくとも $r_t^{(3)}, r_t^{(5)}$ を追跡できれば十分です。最後まで数字を読んだあと、$r_L^{(3)} = 0$ なら `Fizz` の条件を満たし、$r_L^{(5)} = 0$ なら `Buzz` の条件を満たします。両方が成り立つ場合は `FizzBuzz` です。

つまり、FizzBuzz の正解アルゴリズムは、次のような状態更新として書けます。

$$
\begin{aligned}
\text{state}_{t}&=\phi(\text{state}_{t-1}, d_t)\\
\hat{y}&=g(\text{state}_{L})
\end{aligned}
$$

ここで、$\text{state}_t$ は $t$ 桁目まで読んだ時点の内部状態、$\phi$ は数字を読むたびに状態を更新する関数、$g$ は最後の状態からFizzBuzz ラベルを出力する関数です。

この形は、系列モデルにおける scan と非常によく似ています。scan とは、系列を先頭から順に読みながら、内部状態を反復的に更新していく処理です。

この見方をすると、FizzBuzz は単なるお遊びの分類問題ではなく、**ニューラルネットワークが有限状態的なアルゴリズムを獲得し、それをより長い系列へ外挿できるか** を調べる題材になります。

特に、RNN[^1] や GRU[^2] はこのような逐次状態更新を直接モデル化する系列モデルです。GRU では、各時刻 $t$ において入力 $d_t$ に対応する埋め込みを受け取り、隠れ状態を更新します。

$$
\mathbf{h}_t=\mathrm{GRU}(\mathbf{e}_t, \mathbf{h}_{t-1})
$$

ここで、$\mathbf{e}_t$ は数字 $d_t$ の埋め込みベクトル、$\mathbf{h}_t$ は $t$ 桁目まで読んだ時点の隠れ状態です。最後に $\mathbf{h}_L$ を使って4クラス分類を行います。

この構造は、先ほどの剰余更新と対応しているように見えます。もちろん、モデルには剰余そのものを与えません。モデルが学習するのは連続値の隠れ状態であり、剰余状態を明示的に持っているとは限りません。しかし、正解規則が逐次的な状態更新として表現できる以上、GRUのような系列モデルはこの問題に対して自然な候補になります。

さらに、この「左から系列を読みながら状態を更新する」という見方は、Mamba[^3] の selective scan とも発想としては近いです。Mamba は、系列を走査しながら内部状態を更新する State Space Model 系のアーキテクチャであり、入力に応じて状態更新を選択的に変化させます。

ただし、本記事で扱う FizzBuzz と Mamba を同一視しているわけではありません。FizzBuzz の剰余更新は、`mod 3` や `mod 5` によって定まる離散的な状態更新です。一方で、GRU や Mamba が扱う内部状態は、学習されたパラメータによって更新される実数値の状態ベクトルです。したがって、ここでの対応は「同じ計算をしている」という意味ではなく、**系列を順に読み、内部状態を更新し、最後の状態から出力する** という計算形式が似ている、という意味です。

したがって、Mamba のような State Space Model 系アーキテクチャも候補にはなり得ますが、本記事ではまず、より基本的な系列モデルである **GRU** をベースラインとして採用します。整数を 10 進数字列として入力し、Digit Embedding、GRU、Linear classifier によって FizzBuzz ラベルを予測します。

この設定では、モデルが5桁以下の規則を単に記憶するだけでは、6桁、7桁、8桁の整数に対して高い性能を維持することは難しいはずです。もし桁数が増えても性能が維持されるなら、モデルは何らかの形で、数字列に対する再利用可能な状態更新規則を獲得している可能性があります。

逆に、訓練範囲では高精度であっても、6桁以降で性能が崩れるなら、モデルは訓練範囲内のパターンには適合できているものの、剰余規則を獲得できていないと解釈できます。

したがって、本実験の問いは次のように言い換えられます。

:::message
ニューラル系列モデルは、1〜5桁の数字列から FizzBuzz の **状態更新規則** を学習し、6桁、7桁、8桁の数字列へ外挿できるのか。
:::

さらに本記事では、モデルサイズを変化させることで、容量の増加がこの外挿性能にどのように影響するかも検証します。

:::details モデルの詳細
本実験では、整数 $n$ を数値特徴量として直接モデルに入力するのではなく、10進表記の digit 列として扱います。例えば、

$$
12345 \rightarrow [1,2,3,4,5]
$$

のように、整数を左から右へ並んだ digit ID 列に変換します。本実装では、digit $0,1,\ldots,9$ に対して、そのまま digit ID $0,1,\ldots,9$ を割り当てます。

ここで重要なのは、digit ID の整数値を数値特徴量として扱っているわけではないという点です。例えば digit ID $7$ は、「数値としての $7$」をモデルに入力しているのではなく、「digit $7$ に対応する embedding vector を取り出すための ID」です。つまり、digit ID は embedding table を参照するための index として使われます。

したがって、embedding を持つ対象は整数 $1,2,\ldots,99999$ ではありません。embedding を持つ対象は、あくまで digit $0,1,2,\ldots,9$ です。

また、可変長の digit 列を扱うために、padding 用の ID も導入します。本実装では ID の種類を $V=11$ とし、digit ID $0,\ldots,9$ に加えて、padding ID として $10$ を用います。

embedding 次元を $D$、GRU の hidden 次元を $H$、バッチサイズを $B$、最大系列長を $T$ とします。padding を含む入力 ID 列を

$$
\mathbf{I} \in \{0,1,\ldots,10\}^{B \times T}
$$

とします。ここで、$\mathbf{I}$ の各要素 $I_{b,t}$ は、$b$ 番目のサンプルにおける $t$ 番目の ID を表します。

ここでいう Embedding (`nn.Embedding`) は、入力された ID を index として embedding table から対応するベクトルを取り出す lookup 操作です。Embedding table は、学習可能な行列

$$
\mathbf{E} \in \mathbb{R}^{V \times D}
$$

として表されます。

この embedding table の各行が、それぞれの ID に対応する embedding vector です。例えば、

$$
\mathbf{E}_{0,:}, \mathbf{E}_{1,:}, \ldots, \mathbf{E}_{9,:}, \mathbf{E}_{10,:}
$$

は、それぞれ digit ID $0,1,\ldots,9$ と padding ID $10$ に対応する embedding vector です。

ID $i$ に対応する embedding vector を

$$
\mathbf{e}_i = \mathbf{E}_{i,:} \in \mathbb{R}^{D}
$$

と書くことにします。このとき、digit ID 列は embedding table $\mathbf{E}$ を参照することで、

$$
[1,2,3,4,5]
\xrightarrow{\mathbf{E}}
[\mathbf{e}_1,\mathbf{e}_2,\mathbf{e}_3,\mathbf{e}_4,\mathbf{e}_5]
$$

という embedding vector の列に変換されます。

一般的に、各 ID $I_{b,t}$ に対して、対応する embedding vector を取り出すことで、

$$
\mathbf{X}_{b,t,:} = \mathbf{e}_{I_{b,t}}
$$

を得ます。したがって、embedding 後の入力は、

$$
\mathbf{X} \in \mathbb{R}^{B \times T \times D}
$$

となります。

padding 処理についても、具体例で確認します。最大系列長が $5$ の batch 内で整数 $123$ を扱うとします。この場合、元の digit ID 列は

$$
[1,2,3]
$$

です。ただし、batch 内で系列長をそろえるため、右 padding によって padding ID $10$ を末尾に追加します。

$$
[1,2,3]
\rightarrow
[1,2,3,10,10]
$$

この入力 ID 列を embedding table で変換すると、

$$
123
\rightarrow
[1,2,3,10,10]
\xrightarrow{\mathbf{E}}
[\mathbf{e}_1,\mathbf{e}_2,\mathbf{e}_3,\mathbf{e}_{10},\mathbf{e}_{10}]
$$

となります。この例では、実際の digit は $1,2,3$ の3つなので、有効系列長は $\ell=3$ です。

本実装では、右 padding された系列と有効系列長 $\ell$ を用いて `pack_padded_sequence` を適用しています。そのため、GRU は padding 部分を系列として処理せず、有効な digit 列のみを読みます。上の例では、GRU が実際に読む系列は $[\mathbf{e}_1,\mathbf{e}_2,\mathbf{e}_3]$ であり、$[\mathbf{e}_{10},\mathbf{e}_{10}]$ に対応する padding 部分は系列処理から除外されます。

ここからは、padding 部分を除いた有効系列に対する GRU の処理を説明します。各時刻 $t$ における入力を $\mathbf{x}_t$ 、前時刻の hidden state を $\mathbf{h}_{t-1}$ とすると、GRU は次の hidden state $\mathbf{h}_t$ を計算します。それぞれの形状は、

$$
\mathbf{x}_t \in \mathbb{R}^{B \times D},\quad
\mathbf{h}_{t-1} \in \mathbb{R}^{B \times H},\quad
\mathbf{h}_{t} \in \mathbb{R}^{B \times H}
$$

です。

GRU の更新は、概念的には

$$
\mathbf{h}_t = f_{\theta}(\mathbf{x}_t, \mathbf{h}_{t-1})
$$

と書けます。ここで、$f_{\theta}$ は GRU の学習可能パラメータ $\theta$ によって定まる状態更新関数です。

GRU の詳細にはここでは立ち入りませんが、重要なのは、この $\theta$ が時刻ごとに異なるのではなく、全時刻で共有される点です。つまり、5桁の入力に対しては、

$$
\begin{aligned}
\mathbf{h}_1 &= f_{\theta}(\mathbf{x}_1, \mathbf{h}_0) \\
\mathbf{h}_2 &= f_{\theta}(\mathbf{x}_2, \mathbf{h}_1) \\
\mathbf{h}_3 &= f_{\theta}(\mathbf{x}_3, \mathbf{h}_2) \\
\mathbf{h}_4 &= f_{\theta}(\mathbf{x}_4, \mathbf{h}_3) \\
\mathbf{h}_5 &= f_{\theta}(\mathbf{x}_5, \mathbf{h}_4)
\end{aligned}
$$

のように、同じ更新関数を繰り返し適用します。

したがって、6桁の入力に対しても同じ更新関数を追加で適用できます。

$$
\mathbf{h}_6 = f_{\theta}(\mathbf{x}_6, \mathbf{h}_5)
$$

このように、GRU は系列長に対して同じ状態更新関数を反復適用する構造を持つため、5桁までで学習したモデルを、6桁・7桁・8桁の入力にも構造上は適用できます。

ただし、これは外挿性能を保証するものではありません。GRU が単に学習範囲内の digit パターンを記憶しているだけであれば、桁数が増えたときに性能は崩れます。一方で、FizzBuzz の背後にある剰余規則に対応する状態更新を内部的に獲得していれば、桁数が増えても性能を維持できる可能性があります。

実際、10進表記の digit を左から順に読むとき、整数の $m$ による剰余は

$$
r_t^{(m)}=(10r_{t-1}^{(m)} + d_t) \bmod m
$$

という再帰的な更新で計算できます。FizzBuzz では $m=3$ と $m=5$ に対する剰余が分かればよいため、GRU がこのような状態更新を hidden state 内部で近似的に獲得できれば、未知の桁数に対しても汎化できる可能性があります。

最終的に、GRU の最後の hidden state を線形分類器に入力し、FizzBuzz の4クラス分類を行います。

まとめると、本モデルは

$$
\boxed{
\text{digit ID列}
\rightarrow
\text{Embedding}
\rightarrow
\text{GRU}
\rightarrow
\text{Linear classifier}
}
$$

という構成です。
:::

## 4. 未知の桁数への外挿実験

### 4.1 実験条件

本節では、モデル設定と学習設定を示します。本実験では、digit 列を入力とする GRU モデルを用い、**モデル容量の違いが FizzBuzz の桁外挿性能に与える影響** を比較します。

#### 4.1.1 モデル設定

比較するモデルは、`small`、`medium`、`large` の3種類です。各モデルでは、embedding 次元、GRU の hidden 次元、GRU 層数を変更しています。

| Config | Embedding dim | Hidden dim | GRU layers |   総パラメータ数 |
| ------ | ------------: | ---------: | ---------: | --------: |
| `small`  |            16 |         32 |          1 |     5,108 |
| `medium` |            64 |        128 |          2 |   174,788 |
| `large`  |           256 |        512 |          4 | 5,915,396 |

ここで、embedding dim は digit ID を変換する embedding vector の次元数、hidden dim は GRU の hidden state の次元数を表します。GRU layers は、GRU を何層積み重ねるかを表します。

本実験では、これらの設定を変えることで、モデル容量の違いが学習範囲内の性能および桁数方向への外挿性能にどのように影響するかを確認します。

#### 4.1.2 学習設定

各モデルは、以下の共通設定で学習しました。

| 項目            |                  設定値 |
| ------------- | -------------------: |
| random seed   |                   42 |
| epochs        |                  100 |
| batch size    |                1,024 |
| optimizer     |                AdamW |
| learning rate | $1.0 \times 10^{-3}$ |
| weight decay  | $1.0 \times 10^{-4}$ |
| loss function |     CrossEntropyLoss |

最適化には AdamW を用い、学習率（固定）は $1.0 \times 10^{-3}$、weight decay は $1.0 \times 10^{-4}$ としました。FizzBuzz は4クラス分類問題として扱うため、損失関数には Cross Entropy Loss を用いました。

学習設定は **全モデルで共通** とし、比較対象間で変更するのはモデル容量に関するパラメータのみとしました。

### 4.2 実験結果

#### 4.2.1 学習結果

まず、各モデルの学習データに対する性能を示します。

| Config   | Training Acc |         Training Loss |
| -------- | -----------: | --------------------: |
| `small`  |         1.00 | $2.04 \times 10^{-4}$ |
| `medium` |         1.00 | $4.03 \times 10^{-6}$ |
| `large`  |         1.00 | $5.10 \times 10^{-6}$ |

すべてのモデルで Training Acc は $1.00$ に到達しており、5桁以下の FizzBuzz は完全に解けていることが分かります。

この結果は、本実験を考えるうえで重要です。なぜなら、以降で比較する桁外挿性能の差は、単に「学習データを解けたかどうか」の差では説明できないからです。すべてのモデルが学習データに対して完全に適合しているため、この条件では、学習データへの適合と、未知の桁数に対する汎化を分けて観察できます。
> モデルが単に学習範囲内の対応を **記憶** しただけなのか、それとも FizzBuzz の背後にある **剰余規則を獲得しているのか。**

#### 4.2.2 桁外挿性能

学習時に出現しない6桁・7桁・8桁の整数に対する評価結果を示します。

| Config   | Digit length | Test Acc | Macro Precision | Macro Recall | Macro F1 |
| -------- | -----------: | -------: | --------------: | -----------: | -------: |
| `small`  |      6 |   0.9968 |          0.9962 |       0.9971 |   0.9966 |
| `small`  |      7 |   0.9792 |          0.9767 |       0.9794 |   0.9780 |
| `small`  |      8 |   0.9472 |          0.9423 |       0.9439 |   0.9430 |
| `medium` |      6 |   0.9769 |          0.9733 |       0.9716 |   0.9724 |
| `medium` |      7 |   0.9167 |          0.9081 |       0.9004 |   0.9041 |
| `medium` |      8 |   0.8288 |          0.8094 |       0.7950 |   0.8012 |
| `large`  |      6 |   0.9671 |          0.9654 |       0.9660 |   0.9657 |
| `large`  |      7 |   0.8245 |          0.8152 |       0.8154 |   0.8152 |
| `large`  |      8 |   0.6679 |          0.6284 |       0.6249 |   0.6262 |

まず、予想通り、学習時に見ていない桁数へ進むほど外挿性能は低下しました。`small` でも、6桁では Accuracy $0.9968$、Macro F1 $0.9966$ と非常に高い性能を示していますが、8桁では Accuracy $0.9472$、Macro F1 $0.9430$ まで低下しています。学習データは5桁以下であるため、6桁、7桁、8桁と学習範囲から離れるほど、digit 列に対する状態更新を安定して適用することが難しくなっていると考えられます。

一方で、より興味深いのは、**モデルのスケーリングが外挿性能の改善につながらなかった** 点です。当初は、モデル容量を大きくすることで、5桁以下の学習データを記憶するだけでなく、FizzBuzz の背後にある剰余規則をより安定して獲得できる可能性を期待していました。特に `small` と `large` では総パラメータ数に 1000 倍以上の差があり、十分に大きなモデルでは外挿性能が回復する、Double Descent[^4] 的な挙動も期待していました。

しかし、結果は反対でした。`small` が最も高い桁外挿性能を示し、`medium`、`large` とモデル容量を増やすほど、6桁・7桁・8桁のすべてで性能が低下しました。特に `large` は、8桁で Accuracy $0.6679$、Macro F1 $0.6262$ まで低下しており、学習データに対する Training Acc が $1.00$ であることを考えると、学習範囲内への適合と桁外挿性能が大きく乖離していることが分かります。

この結果は、単にモデルを大きくすれば、より一般的な規則を獲得できるわけではないことを示しています。むしろ、容量の大きいモデルほど、5桁以下の学習範囲に対して柔軟に適合できてしまい、その結果として、桁数が増えたときに破綻しやすい解を獲得している可能性があります。

ただし、この表だけでは、どのクラスで性能が崩れているのかは分かりません。そこで次に、混同行列を用いて、桁数が増えたときにどの分類が失敗しているのかを確認します。

[^4]: [Preetum Nakkiran, Gal Kaplun, Yamini Bansal, Tristan Yang, Boaz Barak, and Ilya Sutskever, “Deep Double Descent: Where Bigger Models and More Data Hurt,” arXiv:1912.02292, 2019. ](https://arxiv.org/abs/1912.02292)

混同行列を見ると、誤分類は主に `Number` と `Fizz`、および `Buzz` と `FizzBuzz` の間に集中しています。これは、5の倍数かどうかの判定ではなく、3の倍数かどうかの判定で誤っていることを意味します。

つまり、モデルは末尾 digit に基づく5の倍数規則は比較的安定して外挿できています。一方で、系列全体の digit の和に依存する3の倍数規則は、桁数が増えたときに不安定になっています。

この傾向は、モデルサイズを大きくした場合にも改善されませんでした。むしろ今回の結果では、`small`、`medium`、`large` とモデル容量を増やすほど、また6桁、7桁、8桁と学習範囲から離れるほど、3の倍数が絡む判定の誤りが増えています。

したがって、**100 epoch 時点では** 、モデルスケーリングによって桁外挿性能が改善するというより、**容量の大きいモデルほど5桁以下の学習範囲に強く適合し、未知の桁数に対して3の倍数規則を安定して適用できていない** ように見えます。

ここまでの結果から、すべてのモデルが5桁以下の学習データを解けている一方で、未知の桁数に対する外挿性能には大きな差があることが分かりました。つまり、この差は「学習データを解けたかどうか」ではなく、**「学習の過程でどのような解を獲得したか」** に由来している可能性があります。

同じ Training Acc $1.00$ に到達していても、FizzBuzz の剰余規則に近い状態更新を獲得している場合と、5桁以下の範囲に特化した解を獲得している場合では、桁数が増えたときの性能は異なります。そこで次に、各モデルの学習曲線を確認し、外挿性能の差が学習過程とどのように関係しているのかを見ていきます。

#### 4.2.4 学習曲線

![](https://static.zenn.studio/user-upload/c06f3d347807-20260611.png =500x)
*Fig. `small`, `medium`, `large` の学習曲線*

学習曲線を見ると、Training Acc は最初から滑らかに $1.00$ へ近づくのではなく、しばらく $0.667$ 付近で停滞した後、ある epoch で **急激に** $1.00$ 近くまで上昇しています。

この $0.667$ 付近の plateau は、FizzBuzz のクラス構造を考えると自然に解釈できます。FizzBuzz では、5の倍数判定は末尾 digit が $0$ または $5$ であるかを見るだけでよいため、digit 列から比較的簡単に学習できます。一方で、3の倍数判定は各桁の和の剰余に依存するため、系列全体の情報を保持する必要があります。

ここで、仮にモデルが3の倍数判定をまったく使わず、5の倍数判定だけに基づいて

* 5の倍数でないものを `Number`
* 5の倍数であるものを `Buzz`

と予測したとします。この場合、正しく分類できるのは `Number` と `Buzz` のみです。FizzBuzz のクラス比は、

$$
\text{Number}:\text{Fizz}:\text{Buzz}:\text{FizzBuzz}=8:4:2:1
$$

なので、このショートカット解の Accuracy は

$$
\frac{8+2}{15}=\frac{10}{15}\simeq 0.667
$$

になります。

したがって、学習初期に見られる $0.667$ 付近の plateau は、モデルがまず末尾 digit に基づく5の倍数規則を利用している状態だと解釈できます。この段階では、`Number` と `Buzz` はある程度分類できますが、3の倍数が絡む `Fizz` と `FizzBuzz` は十分に扱えていません。

その後、ある epoch を境に Training Acc が急激に $1.00$ へ上昇しています。これは、モデルが5の倍数判定だけでは解けなかった3の倍数判定を **急に獲得** し、`Fizz` と `FizzBuzz` を正しく分類できるようになったことを示唆しています。

また、この急激な上昇が起きる epoch はモデルによって異なります。つまり、すべてのモデルが最終的には Training Acc $1.00$ に到達しているものの、3の倍数規則を獲得するタイミングはモデルサイズによって異なっています。

このように、学習曲線は単純に滑らかに改善しているのではなく、ある段階で解き方が切り替わるような挙動を示しています。これは、訓練データへの適合が先に進み、その後かなり遅れてテスト性能が改善する **grokking**[^5] を連想させます（ `large` の性能が改善する可能性がある）。

ただし、この学習曲線だけでは grokking と断定することはできません。ここで確認できているのは、あくまで学習データに対する Accuracy の推移です。また、3の倍数規則を学習データ上で解けるようになったことと、それを未知の桁数へ外挿できることは同じではありません。実際、前節で見たように、100 epoch 時点ではすべてのモデルが Training Acc $1.00$ に到達しているにもかかわらず、6桁・7桁・8桁への外挿性能には大きな差がありました。

したがって、grokking 的な挙動として解釈するためには、Training Acc が $1.00$ に到達した後、未知の桁数に対する性能、すなわち6桁・7桁・8桁への外挿性能がどのように変化するかを確認する必要があります。

そこで次節では、学習 epoch 数を変化させ、各時点の重みに対して桁外挿性能を評価します。これにより、**訓練データを完全に解けるようになった後に、外挿性能が遅れて改善するのか** を検証します。

:::details 余談
FizzBuzz を機械学習で解く既存の例では、整数を binary encoding して入力する設定もよく見かけます。おそらく、入力系列長をある程度確保し、系列モデルで扱いやすい問題にする意図があるのでしょう。

ただ、本記事では binary encoding ではなく、10進digit列として整数を扱います。理由は単純で、FizzBuzz は通常、10進表記の整数に対して考える問題であり、特別な理由なしに binary encoding へ変換することにはやや不自然さを感じたからです。そこで本記事では、整数を人間が普段読むのと同じように、10進表記の digit 列としてモデルに入力しています。

結果的に、この設定は学習過程を見るうえでも面白い構造を持っていました。10進表記では、5の倍数判定は **末尾** digit が $0$ または $5$ であるかを見るだけで解けます。一方で、3の倍数判定には digit 列 **全体** の情報が必要です。そのため、モデルにとっては「末尾 digit だけで解ける簡単な規則」と「系列全体を使わないと解けない規則」が同居することになります。

最初からこの挙動を計算して設計したわけではありませんが、結果的にはかなりラッキーでした。素直に10進digit列として扱ったことで、特徴的な学習曲線が現れ、次章での grokking 的な挙動を検証するきっかけにもなりました。

> binary encoding にすれば、5の倍数判定も末尾 digit だけでは解けなくなるため、FizzBuzz はより純粋な「bit 列を scan しながら剰余状態を更新する問題」になります。
:::

[^5]: [Alethea Power, Yuri Burda, Harri Edwards, Igor Babuschkin, and Vedant Misra, “Grokking: Generalization Beyond Overfitting on Small Algorithmic Datasets,” arXiv:2201.02177, 2022. ](https://arxiv.org/abs/2201.02177)

## 5. Grokking 的挙動の検証

### 5.1 実験条件

本章では、学習 epoch 数を変化させ、各時点のモデルに対して桁外挿性能を評価します。具体的には、以下の表の checkpoint におけるモデルを保存し、それぞれについて6桁・7桁・8桁のテストデータで評価します。

評価対象のモデル構成は、前章と同じ `small`、`medium`、`large` の3種類です。学習データ、評価データ、optimizer、損失関数などの基本設定も前章と同一とし、変更するのは学習 epoch 数のみです。

| 項目                | 設定                                     |
| ----------------- | -------------------------------------- |
| モデル               | `small`, `medium`, `large`             |
| 学習データ             | 1桁〜5桁の整数                               |
| 評価データ             | 6桁・7桁・8桁の整数                            |
| optimizer         | AdamW                                  |
| learning rate     | $1.0 \times 10^{-3}$                   |
| weight decay      | $1.0 \times 10^{-4}$                   |
| loss function     | CrossEntropyLoss                       |
| checkpoint epochs | 10, 50, 100, 500, 1000, 5000, 10000 |

この実験の目的は、Training Acc が $1.00$ に到達した後も、桁外挿性能が変化するかを確認することです。もし訓練性能が先に飽和し、その後かなり遅れて外挿性能が改善するのであれば、FizzBuzz においても grokking 的な挙動が生じている可能性があります。

一方で、学習を長く続けても外挿性能が改善しない場合、モデルは5桁以下の学習範囲に対しては適合しているものの、未知の桁数へ適用可能な形では FizzBuzz の規則を獲得できていないと考えられます。

### 5.2 実験結果

以下に、各モデル・各 checkpoint に対するテスト性能の推移を示します。前章で確認したように、いずれのモデルも 100 epoch 以内に訓練精度 100% に到達します。したがって、ここでは **「学習データに完全適合した後、外挿性能がどのように変化するか」** という観点から結果を観察します。

*Fig. `small` の epoch sweep における train および桁外挿テスト性能の推移*
![](https://static.zenn.studio/user-upload/51b345f5ea76-20260612.png)

まず、`small` と `medium` では、epoch を伸ばすことで遅れて汎化性能が改善するような grokking-like な挙動は確認されませんでした。両モデルとも 100 epoch 時点で既に高い外挿性能を示しており、そもそも汎化性能の改善は遅れていなかったので当然の結果です。

一方で、`small` と `medium` の挙動には違いも見られます。`small` では epoch を伸ばすにつれて外挿性能が低下していき、桁が上がるほど明確に悪化しています。これに対して、`medium` では一度性能が低下するものの、その後、持ち直すような挙動が見られました。この差は、モデル容量の違いが長時間学習時の外挿性能に影響している可能性を示しています。ただし、この点については本実験だけで結論づけることはできません。

これに対して、`large` では **grokking-like な挙動** が確認されました。学習データ自体には 100 epoch 以内に完全適合しているにもかかわらず、Test-8 digit の性能は 1000 epoch 以降から改善を始めています。今回の実験では計算時間の都合上 10000 epoch で学習を打ち切りましたが、終盤でも性能はまだ上昇傾向にあり、より長く学習した場合には、さらに性能が伸びる可能性もあります。

また、`large` の Accuracy の推移を見ると、Test-8 digit の性能が上昇する途中に小さなスパイクのような変動も確認できます。この変動が何を意味するのかは明確ではありませんが、訓練データに適合した後も、モデルの内部状態が変化し、異なる解へ移動している可能性を示唆しています。

以上より、`small` と `medium` では明確な grokking-like な挙動は確認されませんでした。一方で、`large` では訓練性能が飽和した後も遠い桁数への外挿性能が改善し続ける特徴的な挙動が見られました。ここでは観察結果の整理にとどめ、この挙動を grokking とどのように関係づけるべきかについては、関連研究を踏まえて次章で考察します。

## 6. 考察

ここまで、モデルサイズごとの外挿性能と、長時間学習による性能変化を確認してきました。特に `large` では、訓練データへ完全適合した後に、外挿性能が遅れて改善する grokking 的な挙動が見られました。

本章では、この結果を既往研究を参照しながら考察します。ポイントは、単に「精度が上がったかどうか」を見るのではなく、訓練データへの適合後に、モデル内部でどのような表現や計算機構が形成され、それがどのように外挿性能として現れるのかを考えることです。

ここでは、二つの視点を分けて整理します。

第一に、Liu らの研究[^6]を参照し、grokking を **表現学習の遅延** として捉えます。すなわち、モデルが訓練データを記憶した後に、タスク構造を反映した内部表現をゆっくり獲得していく、という見方です。

第二に、Nanda らの研究[^7]を参照し、grokking を **記憶回路から汎化回路への移行** として捉えます。すなわち、汎化可能な回路は突然生まれるのではなく、訓練中に徐々に形成され、最後に記憶成分が削られることで外からは突然の汎化として観測される、という見方です。

この二つの視点を組み合わせると、今回 `large` で観察された遅延汎化は、**「訓練データを覚えた後に、桁外挿に有効な剰余構造が内部でゆっくり形成されていった現象」** として解釈できます。

### 6.1 表現学習の観点から見た FizzBuzz Grokking

まず、grokking を表現学習の観点から解析した Liu らの論文を参照します。

@[card](https://arxiv.org/abs/2205.10343)

この論文の中心的な主張は、grokking を「モデルがある時点で突然理解する現象」としてではなく、**訓練データの記憶とは別の時間スケールで、タスク構造を反映した内部表現が形成される現象** として捉える点にあります。

![](https://static.zenn.studio/user-upload/7ae59aac479f-20260613.png)
*同論文より引用。Fig. 1：モジュラー加算を学習する Transformer において、異なる訓練段階で学習された入力埋め込みを可視化したもの。汎化は、埋め込みに構造が出現することと対応している。*

この図では、訓練の初期段階では入力埋め込みに明確な構造が見られません。しかし、汎化が生じる段階になると、埋め込みが円状の構造を形成します。モジュラー加算は周期構造を持つため、入力記号がその周期性を反映した幾何構造を獲得したと解釈できます。

この見方は、今回の FizzBuzz 実験にも自然に接続できます。

FizzBuzz は、出力だけを見ると `Number`、`Fizz`、`Buzz`、`FizzBuzz` の 4 クラス分類です。しかし、その背後にある規則は、

$$
n \bmod 3,\quad n \bmod 5
$$

によって決まります。したがって、より本質的には、

$$
n \bmod 15
$$

に関する周期構造を内部的に扱えるかどうかが重要になります。

つまり、モデルが桁外挿に成功するためには、単に訓練範囲の入出力対応を覚えるだけでは不十分です。digit 列から、3 や 5 に関する **剰余構造** を何らかの形で獲得する必要があります。

この観点から見ると、訓練精度が先に 100% に到達しても、それだけでは十分ではありません。訓練データに完全適合している段階でも、内部表現が FizzBuzz の規則を外挿可能な形で構造化されていなければ、未知の桁数に対しては汎化できません。逆に、学習を継続する過程でそのような構造化表現が形成されると、訓練精度はすでに飽和しているにもかかわらず、テスト精度だけが遅れて改善し始めます。これが grokking 的な挙動として観察されます。

今回の実験では、特に `large` においてこのような遅延汎化に近い挙動が見られました。`large` は 100 epoch 以内に訓練データへ完全適合しているにもかかわらず、Test-8 digit の性能は 1000 epoch 以降に改善を始めました。これは、訓練集合の記憶が先に完了し、**その後に桁外挿へ寄与する内部表現がゆっくり形成されていった** と解釈できます。

さらに重要なのは、grokking が必ずしも不可避の現象ではないという点です。この論文では、学習ダイナミクスを **comprehension**, **grokking**, **memorization**, **confusion** の 4 つの相として整理しています。

![](https://static.zenn.studio/user-upload/006b3ebe68f6-20260613.png)
*同論文より引用。Fig. 7：Transformer における grokking の解析結果。左は、汎化の前後で埋め込み表現の実効次元が変化することを示している。中央は、dropout によって汎化が早まることを示している。右は、decoder の学習率と weight decay によって学習相が切り替わることを示す相図。*

| 相             | 挙動                     |
| ------------- | ---------------------- |
| comprehension | 訓練精度と検証精度が早い段階でともに上がる  |
| grokking      | 訓練精度が先に上がり、検証精度が遅れて上がる |
| memorization  | 訓練精度は上がるが、検証精度は上がらない   |
| confusion     | 訓練精度すら十分に上がらない         |

この整理では、grokking は comprehension と memorization の **間** に位置づけられます。つまり grokking は、汎化能力が神秘的に突然現れる現象というより、**表現学習と記憶の競合の中で生じる遅延相** と見なせます。適切なハイパーパラメータ調整や正則化によって、grokking 相から comprehension 相へ移動できる可能性があります。

:::details 学習相とは何か
ここでいう **相** とは、ハイパーパラメータを変えたときに現れる、学習ダイナミクスの大まかな振る舞いの分類です。
> この「相」という言い方は、物理における相をイメージすると分かりやすいです。たとえば水は、温度や圧力によって氷・水・水蒸気という異なる状態を取ります。同じ H₂O であっても、外部条件が変わるとマクロな振る舞いが大きく変わります。

原論文では、decoder の学習率や weight decay などを変えたとき、モデルの挙動が以下の4つに分かれると整理されています（モデルの詳細に関しては原論文を参照してください）。

ニューラルネットワークは、モデル構造やデータセットが同じであっても、学習率・weight decay・dropout・モデル容量などの条件によって、学習の振る舞いが大きく変わります。

たとえば、decoder が強すぎると、モデルは入力表現をきれいに構造化しなくても訓練データを覚えられてしまいます。この場合、訓練精度は上がるものの、外挿性能は上がりにくく、memorization 相に入ります。

一方で、decoder が弱すぎたり、正則化が強すぎたりすると、訓練データすら十分に学習できず、confusion 相に入ります。

その中間に、表現学習が起きる領域があります。原論文ではこの領域を **Goldilocks zone** と呼んでいます。これは、弱すぎず強すぎず、ちょうどよい条件で表現学習が進む領域です。この領域には comprehension と grokking が含まるとされています。

この議論は、原論文が強調している **「深層学習の統計物理学」** という見方にもつながります。原論文では、低レベルなネットワークダイナミクスを「ミクロ物理」、高レベルなモデル挙動を「熱力学」にたとえています。個々のパラメータや埋め込みがどのように動くかというミクロな視点と、訓練精度・検証精度・汎化・過適合といったマクロな振る舞いを結びつけようとしているわけです。

この視点では、水と同様にニューラルネットワークでも、埋め込みや重みの学習ダイナミクスというミクロな現象が、comprehension・grokking・memorization・confusion というマクロな学習相として現れる、と見ることができます。
:::

今回の `small` と `medium` で明確な grokking が見られなかった理由も、この相図の観点から整理できます。これらのモデルは訓練精度 100% を達成しているため、単純に「容量が足りなかった」とは言えません。むしろ、`small` や `medium` では、モデル容量・最適化・暗黙的正則化のバランスにより、比較的早い段階で外挿に有利な表現へ到達した可能性があります。すなわち、comprehension に近い挙動を示したとも解釈できます。

一方で、`large` は仮説空間が広いため、訓練データを満たす解がより多く存在します。その中には、FizzBuzz の完全な規則を反映した解だけでなく、1〜99999 の範囲では正しく振る舞うものの、桁外挿では崩れる解も含まれるはずです。そのため、`large` では **序盤にそのような記憶的な解に入り、その後、学習を継続する中でより構造的な解へ移動した** と考えれば、今回観察された遅延汎化は自然に説明できます。

よって、今回 `large` で観察された挙動は、grokking 的な遅延汎化と解釈できます。ただし、それは「モデルがある epoch で突然 FizzBuzz を理解し始めた」というより、**訓練集合の記憶とは異なる時間スケールで、桁外挿に有効な内部表現が形成されていった** と見る方が妥当です。

### 6.2 記憶から汎化回路へ：Grokking の内部過程

次に、grokking をモデルの内部機構から解析した Nanda らの論文を参照します。

@[card](https://arxiv.org/abs/2301.05217)

Liu らの論文が「表現がいつ構造化されるか」を捉えていたのに対し、Nanda らの論文は「その構造化された表現が、どのような計算回路として実装されているか」をさらに詳しく調べています。

この論文では、modular addition を解く小規模 Transformer が解析されています。一見すると、モデルは訓練データを記憶したあと、ある時点で突然テスト性能を改善するように見えます。これは典型的な grokking の挙動です。

しかし、Nanda らはモデルの重みと activation をリバースエンジニアリングし、モデルが実際には Fourier 的な回路を学習していることを示しました。具体的には、入力を $\sin$, $\cos$ による周期表現へ写像し、三角恒等式を用いて円周上の回転を合成することで、modular addition を実装していました。

重要なのは、grokking の相転移的な見え方の背後に、より連続的な内部過程が存在するという点です。論文では、この訓練過程を次の三段階に分けて説明しています。

1. **memorization**：訓練データを記憶する段階
2. **circuit formation**：汎化可能な回路が形成される段階
3. **cleanup**：weight decay によって記憶成分が削られる段階

外から train/test accuracy だけを見ると、汎化は突然起こったように見えます。しかし、内部ではその前から汎化回路が徐々に形成されており、最後に記憶成分が取り除かれることで、テスト性能の急激な改善として観測されます。つまり grokking は、突然の汎化獲得というより、**記憶解から汎化回路へと主役が入れ替わる過程** として理解できます。

:::details grokking の段階

![](https://static.zenn.studio/user-upload/cd3441d8d380-20260613.gif)

この動画は、Nanda らによる grokking 解説ページを画面録画したものです。左側には、訓練中の embedding matrix を Fourier basis で見たときの $\sin$, $\cos$ 成分の大きさが示されています。右側には train loss と test loss の推移が示されており、縦の破線は、左側の可視化がどの epoch の重みに対応しているかを表しています。

@[card](https://www.neelnanda.io/grokking-paper)

重要なのは、test loss が急激に低下するよりも前から、左側の周波数成分が徐々に整理されていく点です。訓練初期には、多くの周波数成分が広く使われています。しかし訓練が進むにつれて、不要な周波数成分は弱まり、最終的に少数の key frequencies が強く残るようになります。これは、モデルが訓練データを単に記憶する状態から、modular addition を解くための Fourier 的な汎化回路へと移行していく様子を示しています。

Nanda らは、この過程を大きく三つの段階として整理しています。

1. **memorization**（0k–1.4k epoch）
   まず、モデルは訓練データを記憶します。この段階では train loss は急速に低下しますが、test loss は高いままです。左側の Fourier 成分を見ても、最終的に使われる key frequencies だけが選択的に強くなっているわけではなく、多くの周波数成分が混在しています。つまり、この時点のモデルは訓練集合には適合しているものの、汎化する Fourier 回路をまだ十分には利用していません。

2. **circuit formation**（1.4k–9.4k epoch）
   次に、汎化に使われる回路が徐々に形成されます。外から train loss や test loss だけを見ると、この段階では大きな変化がないように見えます。しかし、内部では Fourier 成分の構造が変化し始めています。最終的に使われる key frequencies が相対的に強くなり、モデルの計算が記憶的な解から Fourier multiplication algorithm へと滑らかに移行していきます。重要なのは、この汎化回路の形成が、test loss の急激な低下よりも前に始まっている点です。

3. **cleanup**（9.4k–14k epoch）
   最後に、weight decay によって記憶的な成分が削られていきます。完成しつつある Fourier multiplication circuit は、訓練データを個別に記憶する解よりも小さい重みでタスクを解けるため、weight decay は不要な記憶成分を減衰させます。この段階で、左側の Fourier 成分はさらに sparse になり、少数の key frequencies が支配的になります。その結果、右側の test loss が急激に低下し、外から見ると grokking が突然起こったように見えます。

この動画が示しているのは、grokking が「ある epoch で突然、汎化能力を獲得する」現象ではないということです。むしろ、内部では汎化回路が連続的に形成されており、その後 weight decay によって記憶成分が取り除かれることで、test loss の急激な低下として観測されます。

したがって、grokking は

$$
\boxed{
\text{memorization}
\rightarrow
\text{circuit formation}
\rightarrow
\text{cleanup}
}
$$

という過程として理解できます。外から見える相転移的な振る舞いは、内部で進行していた回路形成と、記憶成分の除去が重なった結果だと考えられます。
:::

この考え方は、本実験の FizzBuzz にも転用できます。

FizzBuzz においては

$$
r_{t+1} = (10r_t + d_t) \bmod 15
$$

の剰余状態遷移を内部的に獲得できれば、すべての桁数に対して FizzBuzz 規則を適用できます。ここで、$d_t$ は左から $t$ 番目の数字、$r_t$ は先頭から $t$ 桁まで読んだ整数を 15 で割った余りです。

ただし、本実験における「汎化回路」は、Nanda らの論文における Fourier multiplication circuit そのものではありません。FizzBuzz の場合には、digit 列を読みながら $n\bmod 15$ を更新する **剰余状態遷移回路** に対応すると考えられます。

この観点から見ると、`large` の挙動は興味深いものです。

`large` は 100 epoch 以内に訓練データへ完全適合していました。つまり、少なくとも訓練範囲 $1,\ldots,99999$ に対しては、正しいラベルを出力できる解に到達しています。しかし、それは必ずしも FizzBuzz の完全な規則を獲得したことを意味しません。5 桁までの範囲だけを正しく分類する解は多数存在し、その中には 6 桁以降へ外挿できないものも含まれるからです。

一方で、`large` では 1000 epoch 以降に Test-8 digit の性能が改善し始めています。この挙動は、訓練データを記憶した後も、内部表現が固定されていたわけではないことを示唆しています。外から見ると train accuracy はすでに 100% であり、学習は終わっているように見えます。しかし実際には、**weight decay の影響を受けながら、訓練範囲に特化した記憶的な解から、より単純で外挿可能な剰余計算回路へと移行していた** 可能性があります。

特に FizzBuzz では、3 の倍数判定と 5 の倍数判定が非対称な難しさを持ちます。10 進表記において、5 の倍数判定は末尾 digit だけで決まります。すなわち、最後の digit が 0 または 5 であれば 5 の倍数です。一方で、3 の倍数判定は末尾 digit だけでは決まらず、各 digit の和、あるいは逐次的な $\bmod$ 3 状態の更新を必要とします。

そのため、学習初期ではモデルはまず比較的局所的な特徴、たとえば末尾 digit に基づく 5 の倍数判定や、訓練範囲に特化したパターンを利用して訓練データを解く可能性があります。その後、より長く訓練されることで、3 周期、あるいは $\bmod 15$ の状態遷移に対応する構造が徐々に形成される可能性があります。Test-8 digit の改善が遅れて現れるなら、それは単なる記憶ではなく、桁数をまたいで再利用可能な剰余計算の成分が強くなっていることを反映しているのかもしれません。

ただし、ここで注意すべきなのは、本実験ではモデル内部の回路をまだ直接リバースエンジニアリングしていないという点です。したがって、「`large` の内部で剰余状態遷移回路が形成された」と断定することはできません。現時点で言えるのは、Nanda らの grokking 解析を踏まえると、`large` における遅延した外挿性能の改善は、単なる偶然の性能変動ではなく、**記憶的な解から汎化しやすい内部表現へと移行する過程** として解釈できる、ということです。

この見方を採ると、今後調べるべき対象も明確になります。単に test accuracy の推移を見るだけでなく、hidden state から

$$
n \bmod 3,\quad n \bmod 5,\quad n \bmod 15
$$

が線形に読み出せるかを調べれば、モデル内部に剰余状態が形成されているかを検証できます。また、桁を 1 つ読ませたときの hidden state の変化が

$$
r_{t+1} = (10r_t + d_t) \bmod 15
$$

に対応しているかを調べれば、単なる分類境界ではなく、digit 列に対する逐次的な剰余計算が実装されているかをより直接的に検証できます。

つまり、Nanda らの論文が modular addition に対して行ったことを FizzBuzz に対して行うなら、見るべき対象は Fourier frequency そのものではなく、hidden state 上に形成される剰余計算回路です。

本実験では、`large` が 100 epoch 以内に訓練データを完全に記憶した後、1000 epoch 以降に Test-8 digit の性能を改善し始めました。この挙動は、FizzBuzz における grokking 的現象を示唆しています。ただし、その本質は「ある epoch で突然 FizzBuzz を理解した」というより、**訓練データに特化した解の背後で、桁外挿可能な剰余計算回路が徐々に形成され、weight decay によって記憶的成分が相対的に弱まっていった過程** として見るのが自然です。

今回の実験は題材が「たかが FizzBuzz」でありながらも、ニューラルネットワークが **記憶** と **規則の獲得** の間でどのように振る舞うのかを観察する、小さな grokking 実験になっていたと考えられます。

[^7]: [Neel Nanda, Lawrence Chan, Tom Lieberum, Jess Smith, Jacob Steinhardt, "Progress Measures for Grokking via Mechanistic Interpretability", arXiv:2301.05217, 2023.](https://arxiv.org/abs/2301.05217)

## 7. さらに遊ぶなら：発展的な実験案

今回の実験は思った以上に面白い挙動を示したので、ぜひ他の誰かにも追試してみてほしいです。特に、次のような方向は面白いのではないかと思います。

* **中間表現の解析**
  モデルが FizzBuzz を「理解した」とは何を意味するのでしょうか。たとえば、hidden state や embedding 空間が、`mod 3` や `mod 5` に対応する構造を持っているのかを可視化すると、学習の進行に伴って内部表現がどのように変化するのかを調べられそうです。

* **正則化 sweep**
  Grokking では、weight decay や dropout などの正則化が、記憶と汎化のどちらを優先するかに大きく影響することが知られています。今回の実験では weight decay を $10^{-4}$ に固定し、dropout は $0.0$ としましたが、これらを系統的に sweep することで、モデルが memorization / grokking / comprehension のどの領域に入るのかを調べられそうです。特に、正則化が弱すぎると訓練データへの記憶に留まり、強すぎるとそもそも訓練データへの適合が難しくなる可能性があります。そのため、両者の中間に、FizzBuzz の周期構造を反映した表現が形成されやすい領域が存在するかもしれません。

* **有限オートマトン的な解に到達するための最適化**
  FizzBuzz は本質的には `mod 15` の有限状態問題なので、仮説空間の中には有限オートマトン的な完全解が存在するはずです。しかし、重要なのは「そのような解が存在すること」と「通常の最適化がその解を選ぶこと」は別問題だという点です。特に、十分に大きいモデルでは、学習範囲内の FizzBuzz を完全に満たす解が多数存在します。その中には、訓練範囲には完全適合するものの、未知の桁数には外挿できない解も含まれているはずです。だとすると、問題はモデル容量だけではなく、最適化がどのような解へ向かうのかにあります。通常の AdamW や SGD が有限オートマトン的な規則解を選びにくいのであれば、補助損失、Curriculum Learning[^8]、あるいは状態遷移そのものを正則化するような仕組みが必要になるかもしれません。

* **内部状態の離散化**
  FizzBuzz の規則は、本質的には `mod 3` と `mod 5`、つまり `mod 15` に対応する有限状態の問題です。その意味では、連続値の hidden state を使うだけでなく、VQ-VAE[^9] のように内部状態を離散的な codebook へ写像するモデルにすると、より有限オートマトンに近い形で規則を獲得できる可能性があります。もし hidden state が 15 個程度の状態へ自然に分離するなら、モデルが FizzBuzz の周期構造をかなり明示的に表現していると言えるかもしれません。

* **Mamba など他の系列モデルでの実験**
  本記事では GRU をベースラインとして用いましたが、Mamba のような State Space Model 系アーキテクチャで同じ実験を行うと、アーキテクチャの違いが学習挙動や桁外挿性能にどう影響するのかを比較できます。特に、系列を左から右へ読みながら状態を更新するという観点では、RNN 系、SSM 系、Transformer 系でかなり違った挙動が出るかもしれません。

* **他の数遊びタスクへの拡張**
  FizzBuzz 以外にも、剰余、桁和、回文数、素数判定、倍数判定など、整数に関する玩具問題はいろいろあります。こうしたタスクをニューラルネットワークに解かせることで、モデルがどのようなアルゴリズム的規則を獲得できるのかを比較できるかもしれません。

* **FizzBuzz で学習したモデルの fine-tuning**
  FizzBuzz を解いたモデルを、別の数遊びタスクへ fine-tuning するとどうなるでしょうか。もし規則に対応する状態更新をある程度獲得しているなら、他のタスクでも学習が早くなる、あるいは少ないデータで汎化する可能性があります。

* **複数の数遊びタスクを同時に解かせる**
  単一の FizzBuzz タスクだけでは、モデルがそのタスクに特化したショートカット解を学習してしまう可能性があります。そこで、FizzBuzz だけでなく、偶奇判定、剰余分類、桁和、特定の倍数判定など、複数の数理タスクを同時に解かせる設定も考えられます。複数のタスクで同じ digit 列表現を共有させれば、個別タスクに閉じた記憶ではなく、整数列を処理するためのより汎用的な内部表現や、再利用可能な状態更新規則が形成されるかもしれません。もしそのようなモデルが未知の桁数にも安定して外挿できるなら、その中間表現がどのような構造を持つのかは非常に興味深い対象になります。

* **ガチで外挿精度 100% を目指す**
  今回の実験では、外挿精度 100% には届きませんでした。しかし、ここまで来ると、逆に本気で 100% を狙う設定も試してみたくなります。モデル容量、weight decay、学習 epoch、optimizer、内部状態の離散化などを組み合わせれば、単に学習範囲で正解するだけでなく、未知の桁数に対しても FizzBuzz 規則を完全に再利用できるモデルに近づけるかもしれません。もっとも、これは単にモデルを大きくすればよいという話ではなさそうです。仮に、完全解が仮説空間に存在していたとしても、通常の最適化がその解を選ぶとは限りません。むしろ、100% 外挿を目指すということは、「モデルが完全解を表現できるか」だけでなく、「最適化がその完全解を選べるか」を問う実験になるのだと思います。

[^8]: [Xin Wang, Yudong Chen, and Wenwu Zhu. “A Survey on Curriculum Learning”, arXiv:2010.13166, 2022.](https://arxiv.org/abs/2010.13166)
[^9]: [Aaron van den Oord, Oriol Vinyals, Koray Kavukcuoglu “Neural Discrete Representation Learning”, arXiv:1711.00937, 2017.](https://arxiv.org/abs/1711.00937)

## さいごに

ここまで来ると、もはや FizzBuzz の話をしているのか、深層学習の汎化の話をしているのか分からなくなってきました。出発点は、ただの FizzBuzz だったはずです。

ただ、それこそが今回の実験で面白かった点でもあります。FizzBuzz は一見すると単なるお遊びですが、入力表現、モデル構造、学習過程、外挿性能をきちんと分けて見ると、思ったよりも深い題材でした。ちゃんと外挿問題として定義した瞬間に、記憶と汎化、表現学習、Grokking、正則化といった深層学習の厄介な側面が一気に顔を出してきます。

少なくとも、ニューラルネットワークが「訓練データを解けること」と「再利用可能な規則を獲得すること」は、やはり別物なのでしょう。

結局、ニューラルネットワークの学習とは何なのでしょうか。勉強すればするほど、ニューラルネットワークの学習は、思っていた以上に複雑で、危うい均衡の上に成り立っているように見えてきます。どうしてこいつは、上手く機能するんでしょうか。

謎は深まるばかりです。

そして、そんなことを FizzBuzz を通して考えることになるとは思いませんでした。

どうやら FizzBuzz はまだまだ味がしそうですね。

たかが FizzBuzz。されど FizzBuzz。
