## はじめに

近年の視覚認識では、ラベル付きデータだけでなく、ラベルなしデータから汎用的な表現を獲得する自己教師あり学習 (Self-Supervised Learning; SSL) が重要になっている。なかでも、画像の一部を隠し、残された情報から欠損領域を復元する **masked image modeling (MIM)** は、Vision Transformer (ViT) と Masked Autoencoder (MAE) の成功によって代表的な事前学習手法になった。

ただし、ViT で成功した MIM を ConvNet にそのまま移植できるわけではない。ViT は画像を patch token の列として扱うため、mask された token を encoder から取り除ける。一方、ConvNet は dense な二次元 feature map と畳み込み演算を前提としており、入力の一部を mask しても、通常の convolution では空間全体に計算が走る。

ConvNeXt V2 の論文が面白いのは、このずれに対して、単に ConvNet に MAE を載せるのではなく、自己教師あり学習の枠組みとアーキテクチャを **同時に設計** し直している点にある。具体的には、ConvNet 向けの masked autoencoder である **FCMAE** と、masked pretraining 中の feature collapse を抑える **Global Response Normalization (GRN)** を組み合わせている。

本記事では、ConvNeXt V2 を単なる高性能 ConvNet ではなく、**ConvNet で masked image modeling を成立させるための設計論** として読む。

## 1. 論文の問い：MAE を ConvNet へ移植できるのか

### 1.1 ConvNet を残す理由

ViT を中心とする transformer 系アーキテクチャは、大規模データと大規模モデルを組み合わせたときに強いスケーリング性能を示す。自己教師あり学習との相性もよい。

一方で、ConvNet が過去のものになったわけではない。ConvNeXt は、depthwise convolution、大きめの kernel、inverted bottleneck 的な構造、Layer Normalization などを取り入れることで、純粋な畳み込みモデルでも transformer 系モデルに近い性能を達成できることを示した。

ConvNet には、ViT とは異なる帰納バイアスがある。局所的な空間構造を扱いやすく、平行移動に対して自然な性質を持ち、階層的に低レベル特徴から高レベル特徴へ表現を作る。この性質は、画像認識において依然として有効である。

したがって、ConvNet で SSL を成立させることには意味がある。論点は「ViT の代替を作ること」ではなく、**ConvNet の帰納バイアスを保ったまま、ラベルなしデータから有用な表現を学習できるか** である。

### 1.2 MAE の本質

MAE は、画像の一部を mask し、見えている patch だけを手がかりに欠損領域を復元する自己教師あり学習手法である。ただし、目的は復元画像をきれいに生成することではない。重要なのは、復元タスクを通じて **encoder に下流タスクで使える表現を学ばせる** ことである。

MAE では encoder と decoder の役割が非対称である。encoder は下流タスクで使う表現を作る本体であり、decoder は pretraining 時に reconstruction loss を定義する補助モジュールである。そのため、encoder を重く、decoder を軽くする設計になっている。

ViT-MAE が成立しやすかった理由は、この非対称設計と ViT の token-based architecture が噛み合っていたからである。ViT では画像を patch token の列として扱うため、mask された patch を encoder の入力から取り除ける。たとえば 75% を mask する場合、encoder は残り 25% の可視 token だけを処理すればよい。mask token は encoder には入らず、decoder 側で初めて追加される。

この設計によって、次の三つが同時に成立する。

* encoder が mask 領域を直接見ない。
* 高い mask ratio によって encoder 側の計算量を削減できる。
* mask token が decoder 側に閉じるため、encoder の入力分布が比較的単純に保たれる。

### 1.3 ConvNet で壊れる前提

同じ MIM を ConvNet に適用すると、MAE が ViT で利用していた前提が崩れる。最大の違いは、ConvNet が画像を token の集合ではなく、dense な二次元 feature map として処理する点にある。

通常の convolution は空間全体に適用される。したがって、入力画像の一部を 0 で埋めたり、mask token に置き換えたりしても、mask された領域を含む dense な入力全体に計算が走る。このため、次の問題が生じる。

* **情報遮断の問題**：畳み込みは近傍を参照するため、mask 領域と可視領域の境界で情報が混ざりうる。
* **計算効率の問題**：ViT-MAE のように、mask した分だけ encoder の計算量が減るわけではない。
* **分布ずれの問題**：encoder に mask token を入れると、pretraining 時には存在する人工的な入力が fine-tuning / 推論時には消える。
* **特徴空間の問題**：ConvNet の特徴チャネルが masked pretraining のもとで自然に分化するとは限らない。

つまり、ConvNet に MIM を適用する難しさは、単に mask の実装が面倒という話ではない。ViT-MAE で成立していた「可視 token だけを encoder に渡す」という構造を、ConvNet の計算形式に合わせて作り直す必要がある。

そのために提案されたのが、Fully Convolutional Masked Autoencoder、すなわち **FCMAE** である。

## 2. FCMAE：ConvNet 向けに MIM を作り直す

FCMAE は、masked image modeling の考え方を保ちながら、それを ConvNet の計算構造に合わせて再設計した framework である。中心的な発想は、mask された画像を「値が欠けた dense image」ではなく、**可視領域だけからなる sparse data** として扱う点にある。

![](https://static.zenn.studio/user-upload/96a85383ed74-20260624.png)
***FCMAE framework.** Sparse convolution に基づく ConvNeXt encoder と、軽量な ConvNeXt block decoder から構成される。encoder は可視画素のみを処理し、decoder は符号化された画素と mask token を用いて画像を再構成する。損失はマスク領域に対してのみ計算される。*

### 2.1 マスク画像を sparse data として見る

通常、画像に mask を適用するときは、mask された領域を 0 で埋めるか、学習可能な mask token に置き換える。しかしこの場合、入力は依然として dense な二次元配列であり、convolution も画像全体に対して行われる。mask された場所にも計算が発生する。

FCMAE はこの見方を変える。mask された領域を inactive site、可視領域を active site とみなし、encoder では active site のみを処理する。つまり、mask 領域を事後的に 0 にするのではなく、そもそも **計算対象から外す**。

これは 3D point cloud で使われる sparse convolution に近い発想である。点群では空間全体に dense な voxel grid を作ると無駄が多いため、点が存在する active site のみを処理する。FCMAE はこの考え方を MIM に持ち込み、ConvNet でも「可視部分だけを encoder に渡す」構造を作っている。

@[card](https://arxiv.org/abs/1711.10275)
> Submanifold Sparse Convolution を 3D 点群・voxel segmentation に応用した論文。LiDAR や RGB-D 由来の点群のように、3D 空間内のごく一部だけが occupied であるデータでは、dense convolution は非常に無駄が多い。この文脈では、Submanifold Sparse Convolution はかなり自然な選択肢になる。

ConvNeXt V2 / FCMAE の原論文においては Submanifold Sparse Convolution が 3D 点群ではなく、masked image modeling に使われている。

つまり、ConvNeXt V2 における Submanifold Sparse Convolution は、単なる高速化テクニックではなく、mask token を encoder に入力する代わりに、マスク領域をそもそも計算しないための仕組みである。

この意味で、ConvNeXt V2 の FCMAE は、ViT-MAE の発想をそのまま ConvNet に移植したものではない。ConvNet の dense sliding window という性質に合わせて、masked image を sparse data として扱い直したところに設計上の工夫がある。

#### 付録： 3×3 カーネルによる Submanifold Sparse Convolution

入力画像を

$$
X\in\mathbb{R}^{H\times W}
$$

とし、画素位置 $(i,j)$ の値を $X_{i,j}$ と書く。

また、非ゼロ、あるいは意味のある画素の集合を active site と呼び、

$$
\mathcal{A}\subset \{1,\dots,H\}\times\{1,\dots,W\}
$$

とする。

$3\times3$ の畳み込みカーネルを

$$
K=
\begin{pmatrix}
w_{-1,-1} & w_{-1,0} & w_{-1,1} \\
w_{0,-1} & w_{0,0} & w_{0,1} \\
w_{1,-1} & w_{1,0} & w_{1,1}
\end{pmatrix}
$$

とする。

通常の畳み込みでは、各画素 $(i,j)$ に対して

$$
\begin{aligned}
Y_{i,j}={}&w_{-1,-1}X_{i-1,j-1} &+{}& w_{-1,0}X_{i-1,j} &+{}& w_{-1,1}X_{i-1,j+1}&+{}&\\
&w_{0,-1}X_{i,j-1}     &+{}& w_{0,0}X_{i,j}     &+{}& w_{0,1}X_{i,j+1}&+{}&\\
&w_{1,-1}X_{i+1,j-1}  &+{}& w_{1,0}X_{i+1,j}  &+{}& w_{1,1}X_{i+1,j+1}
\end{aligned}
$$

を計算する。

一方、**Submanifold Sparse Convolution** では、出力を計算する位置を active site に限定する。すなわち、$(i,j)\in\mathcal{A}$ のときだけ $Y_{i,j}$ を計算し、$(i,j)\notin\mathcal{A}$ の画素は計算しない。

さらに、近傍画素についても active site の値だけを使う。したがって、

$$
Y_{i,j}=\sum_{\substack{a,b\in\{-1,0,1\}\\(i+a,j+b)\in\mathcal{A}}}w_{a,b}X_{i+a,j+b},\qquad(i,j)\in\mathcal{A}
$$

となる。

$(i,j)\notin\mathcal{A}$ の場合は、$Y_{i,j}=0$ とみなすか、実装上はそもそも $Y_{i,j}$ を保存しない。

つまり、Submanifold Sparse Convolution は次のように書ける。

$$
Y_{i,j}=
\begin{cases}
\displaystyle
\sum_{\substack{a,b\in\{-1,0,1\}\\(i+a,j+b)\in\mathcal{A}}}w_{a,b}X_{i+a,j+b}&(i,j)\in\mathcal{A},\\
\qquad 0&(i,j)\notin\mathcal{A}.
\end{cases}
$$

通常の sparse convolution との違いは、出力位置の扱いにある。

通常の sparse convolution では、近傍に active site が1つでもあれば、その位置を新たな active site として出力する。そのため、層を重ねると active site が周囲に広がっていく。

一方、Submanifold Sparse Convolution では、

$$
\mathcal{A}_{\mathrm{out}}=\mathcal{A}_{\mathrm{in}}
$$

である。

つまり、入力で active だった画素にだけ出力を作る。近傍の active 画素から情報は集約するが、inactive な位置に新しい active site は作らない。

したがって、Submanifold Sparse Convolution は **active site 上だけで行う畳み込み** と考えればよい。カーネル重み $w_{a,b}$ 自体は通常の畳み込みと同じように存在するが、実際に計算されるのは、出力位置と入力近傍の両方が active な場合だけである。

:::

### 2.2 Sparse convolution encoder

FCMAE では、encoder として ConvNeXt を用いる。ただし pretraining 時には、通常の dense convolution ではなく sparse convolution として扱う。これにより、encoder は可視領域に対してのみ計算を行い、mask された領域は encoder の計算に参加しない。

この設計の意味は明確である。

* mask 領域や mask token に依存する shortcut を抑えやすい。
* encoder に mask token を入力しないため、pretraining と fine-tuning の入力分布のずれを小さくできる。
* dense convolution と異なり、masking による計算削減が可能になる。

論文では、sparse convolution の有無が fine-tuning accuracy に大きく影響することが示されている。

|               設計 | ImageNet-1K fine-tuning accuracy |
| ---------------: | -------------------------------: |
| w/o Sparse conv. |                             79.3 |
|  w/ Sparse conv. |                             83.7 |

この差は、単に画像を mask して復元させるだけでは不十分であり、**encoder がどの領域を処理するか** を明示的に設計する必要があることを示している。

なお、sparse convolution layer は fine-tuning 時に標準的な dense convolution に戻せる。したがって、sparse convolution は主に pretraining 時の仕組みであり、下流タスクでは通常の ConvNeXt として扱える。この点は実用上も重要である。

### 2.3 軽量 decoder と masked reconstruction

FCMAE では、入力画像を $32 \times 32$ patch 単位で分割し、その 60% をランダムに除去する。ConvNeXt は階層的な構造を持つため、mask は最後の stage の解像度で生成し、それをより細かい解像度へ再帰的に upsample することで、各 stage に対応する mask を構成する。

encoder には ConvNeXt を用い、decoder には **単一の ConvNeXt block** に基づく軽量 decoder を用いる。ここで decoder を過度に強くしないことが重要である。MAE 系の SSL で下流タスクに使いたいのは基本的に encoder であり、decoder は reconstruction loss を定義するための補助モジュールにすぎない。decoder が強すぎると、encoder が有用な表現を学ばなくても復元できてしまう可能性がある。

再構成対象は、元画像を patch-wise に正規化した画像である。損失には mean squared error (MSE) を用い、**mask された patch に対してのみ** 計算する。可視領域はすでに encoder に与えられているため、そこを復元させても学習信号としては弱い。モデルに求めたいのは、見えていない領域を周囲の文脈や画像構造から推測することである。

FCMAE の設計は次のように整理できる。

* 入力画像を $32 \times 32$ patch 単位で分割し、その 60% を mask する。
* 階層的な ConvNet に対応するため、mask を各 stage の解像度へ展開する。
* mask された画像を sparse data として扱う。
* encoder は sparse convolution によって可視領域のみを処理する。
* decoder は encoder 出力と mask token を用いて欠損領域を復元する。
* reconstruction loss は mask 領域に対してのみ計算する。

FCMAE は、ConvNet で MIM を実行するための入力・計算構造を与えた。しかし、ConvNeXt V1 に FCMAE を組み合わせるだけでは、ConvNeXt V1 の強い教師あり baseline を明確に上回るところまでは届かない。

|         設定 | ImageNet-1K fine-tuning accuracy |
| ---------: | -------------------------------: |
| Sup. 100ep |                             82.7 |
| Sup. 300ep |                             83.8 |
|      FCMAE |                             83.7 |

この結果は、FCMAE が有効である一方で、問題が masked autoencoder の設計だけではなかったことを示している。次の論点は、ConvNet 側の特徴空間である。

## 3. Feature collapse：FCMAE だけでは足りない

ConvNeXt V1 に FCMAE を適用すると、masked pretraining 自体は可能になる。しかし論文では、特徴チャネルの多様性が失われる **feature collapse** が観察されている。

![](https://static.zenn.studio/user-upload/67e2aeb27feb-20260624.png)
***Feature activation visualization.** 各 feature channel の activation map を小さな正方形として可視化する。見やすさのため、各可視化では 64 channel を表示している。ConvNeXt V1 モデルは feature collapse の問題を抱えており、これはチャネル間に冗長な activation（dead または saturated neuron）が存在することによって特徴づけられる。*

### 3.1 Activation map から見える collapse

activation map の可視化では、多くの feature map がほとんど反応していない、あるいは飽和したような反応を示している。つまり、各チャネルが異なる情報を担うというより、似た反応や使われていない反応が増えている。

ここでいう feature collapse は、contrastive learning で問題になる「すべての入力が同じ表現になる collapse」とは少し違う。問題は、**チャネル方向の多様性が失われること** である。一部のチャネルが死んだり、複数のチャネルが似たような activation を示したりすることで、モデル容量を十分に使えなくなる。

SSL では、特定のラベル集合に特化した特徴ではなく、下流タスクへ転移しやすい汎用表現を学習したい。そのため、特徴チャネルが十分に分化しておらず、冗長な反応が増えることは問題になる。論文では、この現象が主に ConvNeXt block 内の dimension-expansion MLP 層で見られたと報告している。

### 3.2 Feature cosine distance による定量解析

論文では、チャネル間の特徴多様性を定量化するために feature cosine distance analysis を行っている。

activation tensor を $X\in\mathbb{R}^{H\times W\times C}$ とする。ここで、$H$ と $W$ は空間サイズ、$C$ はチャネル数である。また、$X_i\in\mathbb{R}^{H\times W}$ を $i$ 番目のチャネルの feature map とする。各 feature map $X_i$ を $HW$ 次元のベクトルとみなし、チャネル間の平均 pair-wise cosine distance を次のように計算する。

$$
\frac{1}{C^2}\sum_i^C\sum_j^C \frac{1-\cos(X_i,X_j)}{2}
$$

この値は、チャネル同士がどれだけ異なる activation pattern を持つかを表す。cosine distance が小さいほど、多くのチャネルが似た反応をしており、feature redundancy が大きいと解釈できる。

![](https://static.zenn.studio/user-upload/b1761794746b-20260624.png)
***Feature cosine distance analysis.** アーキテクチャによって総層数が異なるため、normalized layer index に対して distance value を plot している。*

図で重要なのは、**FCMAE で事前学習された ConvNeXt V1 が、層が深くなるにつれて feature diversity を大きく失っている** 点である。教師あり学習された ConvNeXt でも最終層付近では特徴が分類に必要な方向へ集約されるが、FCMAE 事前学習ではより広い範囲で冗長化が見られる。

対照的に、MAE で事前学習された ViT では feature diversity が比較的保たれている。これは、ViT-MAE が MIM と構造的に噛み合っていることと整合的である。

この解析により、問題の焦点が明確になる。FCMAE は ConvNet に MIM を適用するための入力・計算構造を与えた。しかし ConvNeXt V1 のままでは、masked pretraining 中に特徴チャネルの多様性が失われやすい。したがって、ConvNet で MIM を成立させるには、**アーキテクチャ内部の特徴応答** も設計する必要がある。

そこで導入されるのが、**Global Response Normalization (GRN)** である。

## 4. GRN：特徴チャネルを競合させる

GRN の目的は、チャネル間の feature competition を強め、feature diversity を維持することである。言い換えれば、各チャネルが似た反応に潰れることを防ぎ、それぞれのチャネルが異なる特徴を担いやすくするための仕組みである。

![](https://static.zenn.studio/user-upload/4109b4bd1a02-20260624.png)
***ConvNeXt Block Designs.** ConvNeXt V2 では、dimension-expansion MLP layer の後に GRN layer を追加し、冗長になる LayerScale を削除する。*

### 4.1 GRN はどこに入るのか

GRN は、ConvNeXt block の dimension-expansion MLP layer の後に挿入される。この位置は、feature collapse が主に MLP 層で観察されたことと対応している。

ConvNeXt block では、まず depthwise convolution によって空間的な情報を処理し、その後 MLP 的なチャネル変換によって特徴を混合・拡張する。この dimension-expansion 部分は表現力を高める一方で、masked pretraining のもとでは一部のチャネルが冗長化しやすい。

GRN は、この高次元化された特徴に対して、各チャネルの global response を比較し、相対的にどのチャネルが強く反応しているかを利用して feature map を調整する。これにより、チャネル間に競合関係が生まれ、似たようなチャネルばかりが増えることを抑える。

ConvNeXt V2 では、GRN の導入により ConvNeXt V1 で用いられていた LayerScale は不要になり、削除されている。変更は比較的小さいが、masked pretraining 中に観察された feature collapse という具体的な問題に対応している。

以下の形がよいです。**数式と実装を一致させるため、正規化は Algorithm 1 に合わせて「チャネル平均 response で割る」形にしています。** 原論文本体の式は総和で割っていますが、Algorithm 1 とは定数倍 $C$ だけ違います。記事ではその点を明記した方が安全です。

### 4.2 GRN は何をしているのか

GRN の処理は、大きく三段階に分けられる。

1. 各チャネルの global response を集約する。
2. チャネル間で response を正規化する。
3. 正規化された response を使って元の feature map を補正する。

入力特徴を $X\in\mathbb{R}^{H\times W\times C}$ とし、$X_i\in\mathbb{R}^{H\times W}$ を $i$ 番目のチャネルの feature map とする。

まず、GRN は各チャネルの空間方向の Frobenius norm を global response として集約する。

$$
g_i=\|X_i\|_F=\sqrt{\sum_{h=1}^{H}\sum_{w=1}^{W}X_{h,w,i}^{2}}
$$

これを全チャネルについて並べると、global response vector は次のように書ける。

$$
\mathcal{G}(X)
=
\begin{bmatrix}
g_1\\
g_2\\
\vdots\\
g_C
\end{bmatrix}
$$

次に、各チャネルの response を、全チャネルの response と比較する。

$$
\mathcal{N}(\mathcal{G}(X))_i=\frac{g_i}{\frac{1}{C}\sum_{j=1}^{C}g_j+\varepsilon}
$$

ここで、$\varepsilon$ はゼロ除算を防ぐための小さな定数である。

これは、$i$ 番目のチャネルが、全チャネルの平均と比べて相対的にどの程度強く反応しているかを表す。絶対的な大きさではなく、チャネル間の相対的な response を見る点が重要である。

最後に、この正規化された response を元の feature map に掛ける。

$$
\tilde{X}_i=\mathcal{N}(\mathcal{G}(X))_i X_i
$$

ここで、$\mathcal{N}(\mathcal{G}(X))_i$ は scalar であり、空間方向に broadcast されて $X_i\in\mathbb{R}^{H\times W}$ に掛けられる。

実際の GRN layer では、learnable parameter である $\gamma,\beta\in\mathbb{R}^{C}$ を加え、さらに residual connection を持たせる。チャネルごとに書けば、出力は次のようになる。

$$
X^\text{out}_i=X_i+\gamma_i\tilde{X}_i+\beta_i\mathbf{1}_{H\times W}
$$

ここで、$\gamma_i$ と $\beta_i$ は scalar であり、空間方向に broadcast される。$\gamma$ と $\beta$ は初期値を 0 とするため、初期状態では GRN は identity function として振る舞う。その後、学習の中で各チャネルの response を補正するように働く。

GRN は、Batch Normalization (BN) や Layer Normalization (LN) のように単に分布を整える層というより、チャネル間の相対的な反応を使って **feature competition を導入する層** と見る方が分かりやすい。

:::message alert
なお、本記事では原論文の記法を、次元と実装との対応が分かりやすい形に書き換えている。原論文では各チャネルの response を L2 norm として書いているが、ここでは $X_i\in\mathbb{R}^{H\times W}$ を feature map として扱うため、空間方向の Frobenius norm $\|X_i\|_F$ として明示している。

また、原論文の本文式では分母を全チャネル response の総和として書いているが、PyTorch-like pseudocode では `gx.mean(dim=-1)` が使われている。本記事では実装との対応を優先し、全チャネルの平均 response で割る形で記述した。
:::

:::details GRN PyTorch Implementation

```python: GRN
from __future__ import annotations

import torch
from torch import Tensor, nn


class GRN(nn.Module):
    """Global Response Normalization for channels-last feature maps.
    This implementation follows the PyTorch-like pseudocode in the
    ConvNeXt V2 paper.
    Parameters
    ----------
    num_channels:
        Number of feature channels.
    eps:
        Small constant for numerical stability.
    Notes
    -----
    The expected input shape is ``(B, H, W, C)``, i.e. channels-last format.
    """

    def __init__(self, num_channels: int, eps: float = 1e-6) -> None:
        super().__init__()

        if num_channels <= 0:
            raise ValueError(f"num_channels must be positive, got {num_channels}.")
        if eps <= 0.0:
            raise ValueError(f"eps must be positive, got {eps}.")

        self.num_channels = num_channels
        self.eps = eps

        # Initialized to zero so that GRN starts as an identity function.
        self.gamma = nn.Parameter(torch.zeros(1, 1, 1, num_channels))
        self.beta = nn.Parameter(torch.zeros(1, 1, 1, num_channels))

    def forward(self, x: Tensor) -> Tensor:
        """Apply GRN.
        Parameters
        ----------
        x:
            Input tensor of shape ``(B, H, W, C)``.
        Returns
        -------
        Tensor
            Output tensor of shape ``(B, H, W, C)``.
        """
        if x.ndim != 4:
            raise ValueError(f"x must be a 4D tensor of shape (B, H, W, C), got {x.shape}.")
        if x.shape[-1] != self.num_channels:
            raise ValueError(
                f"Expected {self.num_channels} channels, got {x.shape[-1]}."
            )
        # gx: (B, 1, 1, C)
        gx = torch.linalg.vector_norm(x, ord=2, dim=(1, 2), keepdim=True)
        # nx: (B, 1, 1, C)
        nx = gx / (gx.mean(dim=-1, keepdim=True) + self.eps)
        return x + self.gamma * (x * nx) + self.beta
```

ablation では、global feature aggregation、normalization function、residual connection の有無、他の normalization layer や gating method との比較が検討されている。重要なのは、GRN が単なる「正規化層の追加」ではないことだ。Frobenius norm による global response、チャネル間での相対化、residual connection 付きの補正という組み合わせが、masked pretraining における feature diversity の維持に効いている。

また、GRN は pretraining と fine-tuning の両方で必要とされる。pretraining 時だけ使って fine-tuning 時に取り除くと性能は下がる。逆に、pretraining 時には使わず、fine-tuning 時だけ追加しても十分には機能しない。つまり、GRN は fine-tuning 用の後付け補正ではなく、事前学習で形成される特徴空間そのものに関わる。

### 4.3 Co-design：FCMAE と GRN の組み合わせ

ConvNeXt V2 の中心的な主張は、FCMAE と GRN が単独ではなく、**組み合わせて効く** という点にある。

![](https://static.zenn.studio/user-upload/5f38fd5956c7-20260624.png)
***Co-design matters.** アーキテクチャと学習フレームワークを共同設計し、それらを組み合わせて用いると、masked image pre-training は ConvNeXt に対して有効になる。*

FCMAE は、ConvNet で MIM を行うための学習フレームワークである。mask された画像を sparse data として扱い、encoder に **可視領域のみ** を処理させる。一方、GRN は、masked pretraining 中に **特徴チャネルが冗長化することを抑える** アーキテクチャ修正である。

ConvNeXt V1 に FCMAE を適用すると、一定の改善は得られる。しかし、それだけでは改善は限定的である。GRN を導入した ConvNeXt V2 も、教師あり学習の設定では効果が比較的小さい。大きな改善が現れるのは、FCMAE と GRN を組み合わせたときである。

この結果から分かるのは、SSL の性能が pretext task だけで決まるわけではないということである。同じ MIM であっても、ViT と ConvNet では適した実装が違う。ViT では token を間引く設計が自然に成立する。一方、ConvNet では sparse convolution によって可視領域のみを処理する工夫が必要になる。さらに、ConvNet 内部では feature collapse が起こりやすく、それを抑えるために GRN のような仕組みが必要になる。

したがって、ConvNeXt V2 は単なる ConvNeXt の改良版ではない。**FCMAE という自己教師あり学習フレームワークと組み合わせることを前提に、ConvNeXt を再設計したモデルである。**

## 5. スケーリングから見える Co-design の有効性

ConvNeXt V2 + FCMAE の有効性は、単一のモデルサイズに限られていない。論文では、Atto から Huge まで、複数のモデルサイズに対して FCMAE 事前学習を適用し、教師あり baseline や ConvNeXt V1 と比較している。

![](https://static.zenn.studio/user-upload/ee138c2af42a-20260624.png)
***ConvNeXt V2 model scaling.** FCMAE を用いて事前学習された ConvNeXt V2 モデルは、幅広いモデルサイズにわたって、従来バージョンよりも大幅に優れた性能を示す。*

ここで重要なのは、改善が特定のモデルサイズにだけ現れているわけではない点である。FCMAE で事前学習された ConvNeXt V2 は、広いモデルスケールにわたって ConvNeXt V1 を上回っている。これは、提案手法が偶然あるサイズにだけ効いたのではなく、モデルを大きくしても一貫して機能する設計であることを示している。

また、改善は ImageNet 分類だけに閉じていない。論文では、COCO における物体検出・インスタンスセグメンテーション、ADE20K における semantic segmentation でも評価されている。自己教師あり事前学習の目的は、reconstruction loss を下げることではなく、下流タスクへ転移しやすい表現を得ることである。その意味で、分類以外のタスクでも改善が見られることは、FCMAE + ConvNeXt V2 によって得られた表現が一定の汎用性を持つことを示している。

この論文から得られる示唆は、次の三点に整理できる。

**第一に、ConvNet で MIM を行う場合、mask token を入力に入れるだけでは不十分である。**
ViT-MAE では、mask された patch を encoder から取り除き、可視 token だけを処理できる。しかし、ConvNet は dense な feature map と convolution を前提とするため、同じ設計をそのまま再現できない。mask 領域からの情報漏洩、mask token による pretraining / fine-tuning 間の分布ずれ、masking による計算削減の難しさを考える必要がある。FCMAE は、mask された画像を sparse data として扱い、encoder が可視領域のみを処理することで、この問題に対処している。

**第二に、reconstruction loss だけでは表現学習の良し悪しは判断できない。**
Masked autoencoder の目的は、きれいな画像を復元することではなく、下流タスクに有用な encoder 表現を学習することである。decoder が強すぎれば、encoder が十分な表現を学ばなくても復元 loss は下がりうる。また、loss が改善していても、内部では feature collapse が起きている可能性がある。したがって、SSL の評価では、pretext task の loss だけでなく、fine-tuning 性能、転移性能、特徴空間の多様性を合わせて見る必要がある。

**第三に、SSL では「手法を選ぶ」だけでなく、「学習信号とモデル構造の噛み合わせ」を見る必要がある。**
MAE や SimMIM のような masked image modeling 系では、mask の単位、encoder に見せる情報、decoder の役割が性能を大きく左右する。一方、DINO のような self-distillation 系や、MIM と self-distillation を組み合わせる iBOT では、teacher-student の設計、view の作り方、feature collapse の回避が重要になる。したがって、SSL objective はアーキテクチャから独立に効くものではない。入力の扱い、mask や augmentation の処理、encoder-decoder あるいは teacher-student の役割分担、特徴空間の挙動まで含めて、手法とモデルを一体として評価する必要がある。

この観点から見ると、ConvNeXt V2 + FCMAE の価値は、単に ConvNet の精度を上げたことではない。むしろ重要なのは、ConvNet に masked image modeling を適用する過程で、**学習フレームワークとアーキテクチャを同時に設計し直す必要がある** ことを示した点にある。ConvNeXt V2 論文タイトルにおける **Co-design** とは、まさにこの関係を指している。

ConvNeXt V2 + FCMAE は、「ConvNet に MAE を載せたらどうなるか」という単純な実験ではなく、ViT で自然に成立していた MIM の前提を ConvNet 上で設計し直した試みである。その意味で、この論文は ConvNet の再評価であると同時に、自己教師あり学習をアーキテクチャから切り離して考えることの難しさを示しているようにも思う。少なくとも、SSL objective はモデル構造から独立に機能するものではなく、入力表現、計算構造、特徴空間の性質と結びつけて設計する必要があると感じた。
